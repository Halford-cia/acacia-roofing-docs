## Stage 3 Brain Coverage Workflow ‚Äî Rejected (2026-05-06)

Built `proposeCoverage_` / `applyCoverage_` / `dispatchCoverageTags_` + Brain prompt addendum across ten local commits. Five rounds of prompt iteration could not get Brain to reliably emit `PROPOSE_COVERAGE` tags ‚Äî model-level limit confirmed via FILES IDENTICAL deploy verification + byte-level prompt audit. Brain consistently asked for "Job ID" (no such thing in coverage), pre-listed all of a rep's outstanding leads, asked which specific rows to mark, and hallucinated $ amounts from prompt context. Even structural primacy (COVERAGE WORKFLOW moved to top of basePrompt) + MENTAL MODEL OVERRIDE framing + WRONG-vs-RIGHT worked examples + explicit HARD RULES could not override the model's "rep closed a job, deduct money, confirm before writing" prior.

**Real architecture insight:** commission requests are simultaneous with coverage events, generated by office, never skipped, with structured lead lists. Coverage should be a SIDE EFFECT of commission generation, not a separate Brain workflow. NL parsing of financial operations is the wrong shape; structured form input is the right shape.

**Path forward:** the Acacia Admin Portal (see below) ‚Äî admin tools shell + rep self-service shell, with coverage as a side effect of commission generation. Brain stays as conversational / query layer (read-only, information-seeking). Long-term goal: canonical Job entity with all tools as interfaces over it.

**Reusable handlers from Stage 3** (kept conceptually, will be re-implemented as portal services in Stage A Commit 2 ‚Äî clean re-implementation, no commit cherry-pick):
- Brain Audit tab + `appendAuditLog_` schema (originally commit `95f8ece`)

- `applyCoverage_` pre-write audit pattern + two-tier idempotency (originally `908d4d9`)

The Stage 3 implementation stack itself was hard-reset away from local `main` on 2026-05-07 prior to starting the portal build (no GitHub push had ever happened, so origin/main was unaffected). The `job-submittal` Apps Script project still has the deployed-but-dead Stage 3 code (handlers + dispatcher + COVERAGE WORKFLOW prompt section) ‚Äî production behavior unchanged because Brain doesn't reliably emit the tags. ### Service Layer Pattern (Commits 2a + 2b)

Two-tier service stack:
- **Foundational services** (Commit 2a): `SheetSchema`, `ScriptCache`, `BrainAuditService`, `RepsService`. No cross-dependencies among themselves except `BrainAuditService` ‚Üí `SheetSchema`. Provide column-mapping, cache, audit-log, and rep-lookup primitives.
- **Operational services** (Commit 2b): `JobsService`, `CostsService`. Depend on the foundational layer. Expose business-shaped reads/writes for Master Tracker + Rep Job Costs.

Conventions (uniform across all services):
- **Service-as-namespace:** `const ServiceName = { method: function(...) {...}, _privateHelper: function(...) {...} };` Public methods non-underscored; private helpers leading-underscored.
- **Universal response shape:** `{ success: true, ...payload }` on success; `{ success: false, error: 'message' }` on failure. Mutation methods that may partial-apply also return `partialApply: true` with row pointers and an audit-row reference for reconciliation.
- **Schema-defensive:** every read calls `SheetSchema.getColumns(spreadsheetId, sheetName, requiredColumns)`, which throws on missing required columns. Services catch internally and surface as `success: false` at the public boundary ‚Äî callers never see throws.
- **Pre-write audit pattern:** mutating methods (`updateJobFields`, `applyCoverage`, `addCost`) write the audit row BEFORE touching data rows. Audit-write fail aborts cleanly with no row mutations. Audit-succeed / row-write fail surfaces as `partialApply` with the audit row pointer for manual reconciliation. Direct port from the Stage 3 reusable handler (`908d4d9`).

### View Layer Pattern (Commit 3)

- **Routes.js holds per-view server handlers.** One top-level function per nav slug (e.g. `getMyJobsHtml`) so `google.script.run` can call them directly. Shared helpers live on the `Routes` namespace (`Routes._formatDate`, `_formatCurrency`, `_slugifyStatus`, `_renderErrorHtml`).
- **View-handler convention:** `getCurrentUser_` first ‚Üí role gate (REP-only / ADMIN-only) ‚Üí service call ‚Üí projection to view shape ‚Üí `HtmlService.createTemplateFromFile + evaluate` ‚Üí return rendered HTML string.
- **Service / view boundary responsibility:** services (`JobsService._rowToJob`, `CostsService._rowToCost`) return ALL columns from spreadsheet rows. The view layer is responsible for filtering admin-only fields out of rep responses. NEVER pass raw service responses to the client without per-role projection.
- **Template escaping:** all interpolation uses `<?= %>` (HTML-escaped). `<?!= %>` is reserved for trusted `include_()` calls only ‚Äî never for data fields.
- **Pattern A:** server-rendered HTML fragments returned via `google.script.run` and swapped into `#view-pane.innerHTML`. No client-side templating; no JSON contract between client and server. Trade-off: simpler than Pattern B (JSON + client render), at the cost of larger payloads and round-trip per nav. For an internal tool with low traffic, the simplicity wins.

### SPA Pattern (Commit 3)

- **ClientSpa.html:** IIFE-wrapped `<script>` included once at the end of `Index.html`'s ADMIN/REP branch (NOT the UNAUTHORIZED branch ‚Äî deny page has no nav).
- **Sandbox-safe constraints:** `getElementsByClassName` (not `querySelectorAll`), `addEventListener` (not inline `onclick=`), string concatenation (not template literals), `escapeHtml()` for any client-side string before `innerHTML`.
- **View-handler routing:** the IIFE knows which server function to call for each nav slug via inline `if/else` (acceptable while the slug count is small; convert to a static `VIEW_HANDLERS` map if it grows past ~6).
- **Welcome-content lifecycle:** `#welcome-content` removed from DOM on first nav click (idempotent guard via `welcomeContent && welcomeContent.parentNode`); `welcome-bar` (the auth/role/version meta strip) is persistent above `#view-pane`.

### ai-lead-helper bug-fix history ‚Äî 2026-05-01 migration cleanup

Context for why the ai-lead-helper code looks the way it does. Surfaced during the migration from acaciaroofingleads@gmail.com ‚Üí henry@acaciaroofs.com (fix for cross-org auth/permission issues).

- **Loop poisoning fix.** `checkBadLeadFormSubmissions` was written so that any throw inside the row loop bubbled to the outer try/catch, which silently logged and exited ‚Äî the offending row never stamped Processed=Yes, so the same row poisoned every 30-min run. 12+ days of bad-lead reports queued up before this was caught (root cause was an empty `LEADS_CONFIG.ADMIN_CC` recipient making `GmailApp.sendEmail` throw `no recipient`). **Fix:** per-row try/catch wraps each iteration; rows that fail stamp `Error: <msg>` (truncated to 80 chars) so a single bad row can't block the whole pass. Skip-check was loosened from `=== 'Yes'` to truthy so Error rows don't infinite-retry. Same guard added to `onNewLeadSubmitted`'s missing-rep-email path.
- **Email flood fix.** When the backlog drained, the no-match path fired one email per unmatched row ‚Äî 24 emails in 30 seconds tripped Gmail's daily quota (`Service invoked too many times for one day: email`). **Fix:** rows accumulate into `unmatched` and `reversals` arrays during the loop; a single summary email goes out per run via `sendBadLeadSummaryEmail_()`. Per-row sends were removed entirely. The `LEADS_CONFIG.ADMIN_CC` empty-guard is still honored ‚Äî no recipient configured, no email sent.
- **Match logic fix.** `matchLeadByAddress` had two compounding bugs: (1) it filtered to `status === 'Pending'` before any match attempt, so every Good-aged lead was unreachable (and reps file reports late), and (2) the word tokenizer ran a regex against an already-stripped string so it always produced one giant concatenated "word", making the `bestScore >= 2` threshold structurally unreachable. Together these dropped 24 of 25 reports as no-match. **Fix:** two-pass match ‚Äî exact normalized match first across all statuses, then street number + ‚â•1 significant name token, preferring Pending row but falling back to non-Pending. Stopwords were narrowed to **pure street-type abbreviations only** (rd, dr, ct, st, ln, ave, blvd, pl, cir, pkwy, apt, unit, suite, ste), directionals (n/s/e/w), and state (tx/texas). Long forms (drive, road, court, ...) and Austin-area city names (austin, round, rock, cedar, park, leander, pflugerville, georgetown) are **NOT** stopwords ‚Äî bias is toward more matches because false-miss cost (lost lead credit, real money) >> false-match cost (manual review via summary email).
- **Status reversal flow.** When a previously-Good lead is flipped to Bad, the run pushes to a `reversals` array which gets a dedicated `=== Status Reversals ‚Äî Manual Action Required ===` section in the summary email, listing customer/address/rep/prev-status/reason. This signals that a Rep Job Cost row was already written to the Master Tracker by `handleGoodLead()` and needs manual reversal ‚Äî see Manual Action Items at the top of CLAUDE.md for the running list.
- **Bad-already path.** Duplicate bad-lead reports for an already-Bad lead don't re-queue a credit request (avoids double-charging the lead company). Log line includes both the existing `Reported` cell value and the new rep/reason for audit trail.
- **Previous Script ID (deprecated 2026-05-01).** Old project ID `18lNBsw_3Jt6teoI9EBMNh0_hbaMoRrDrP4kJBGxkJA-dss6VkDqZT_pN` ‚Äî all triggers deleted, project archived as backup at the migration. Source backup retained in this repo at `ai-lead-helper-old/`.

### bad-lead-app / ai-lead-helper Split ‚Äî full rationale (2026-05-12)

- **Problem:** ai-lead-helper deploys as USER_ACCESSING (so bulk uploader emails route through Josh's acaciaroofingleads@ identity). This means any web app hosted in the project runs as the rep on submit ‚Äî reps need write access to Master Tracker and LEADS TRACKER for the bad-lead path. Reps can't have that access (sensitive financial data).

- **Trade-off:** Some helpers are duplicated across projects (see "Known Duplicated Functions" section below). Drift risk accepted given small surface area.
- **Coordination:** Cross-project state lives in shared spreadsheets (Bad Lead Reports tab is the credit-email handoff point; LEADS TRACKER is read by bad-lead-app and written by both projects on different paths).
